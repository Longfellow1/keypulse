# Claude 评审意见

日期：2026-04-27
评审人：Claude (Opus 4.7)
范围：产品方向、技术实现、远期发展

## 产品方向

### 同意：定位判断正确

"工作记忆编译器"避开了三个红海——编辑器（Logseq/SiYuan/AFFiNE）、RAG 聊天（Khoj/AnythingLLM）、always-on lifelog（Rewind/screenpipe）。竞品调研的结论有判断力：最高匹配度在 agent 记忆层（Basic Memory / IWE / Atomic），最该避开的是编辑器。

Karpathy wiki 思路的映射也到位——收集原始资料、蒸馏成可读 Markdown、保留来源链接、随时间编译更高层页面。这个管线定义比"又一个 RAG 工具"清晰得多。

### 顾虑：产品 loop 还没闭环

现在能写每日笔记到 Obsidian，但"用户看了笔记之后呢？" checkbox 反馈机制设计文档里写了，代码里实现了吗？如果用户只是读、不反馈、不纠正，系统的长期价值会衰减。cold start 14 天设计很细致，但第 15 天用户留存靠什么——目前看不到答案。

### 建议

- 在 Phase 1（Evidence notes）做扎实后，跳过 Phase 3-4-5 的理想化分层，直接做"用户能用自然语言问今天做了什么"的最简实现。验证需求比完善管线重要。
- Markdown-first 且 Obsidian 作为旗舰集成的策略是对的——不要把自己绑死在一个编辑器上。
- 第一目标用户画像需要明确。目前看最自然的是"已经用 Obsidian 的技术工作者"，但这个群体有多大有待验证。

## 技术实现

### 同意：隐私债务必须最先清

Codex 评审发现的 5 个问题（keyboard chunk 原文落盘、metadata 未脱敏、隐私窗口排除未实现、local-first 可 fallback 云端、scrub 命令不可用）都是结构性风险。RAG 功能可以被复制，隐私信任一旦破了，本地优先的定位就站不住。

Phase 0（信任加固）排第一的判断完全正确。

### 顾虑：叙事 pipeline 设计过度，验证不足

7 动机骨架、双遍 LLM、迭代置信度更新——这套架构在纸面上优雅，但 v0 还没跑通。最应该先验证的假设是："算法构架 + LLM 措辞"的产出，真的比"直接把清洗后的 fragments 丢给 LLM 写总结"好多少？

这个 A/B 测试没做之前，整个 skeleton 设计都是基于直觉的。建议：

- 用现有 narrative_v2 的输出做 baseline。
- 跑 skeleton v0 的输出做对比。
- 找 3-5 个真实用户（哪怕是朋友）读两边输出，告诉我哪个更有用。
- 如果 skeleton 没有显著更好，砍掉复杂度。

### 顾虑：测试覆盖了功能，没覆盖体验

333 个测试通过是好事，但都是单元测试 + 集成测试。没有端到端测试覆盖"采集一天数据 → 生成每日笔记 → 用户读到内容"这个核心 loop。建议补一个 e2e test，用录制好的 event stream 跑完整管线，验证输出 Markdown 的结构和内容质量。

## 远期发展

### Phase 3-5 的路线图太远

对于 solo 项目，12-18 个月的路线图弊大于利。Phase 4（Agent context / MCP read-only）和 Phase 5（Controlled write-back）的设想是合理的，但现在规划细节是浪费时间——前面验证的结果会改变后面的一切。

建议只保留两个里程碑：
1. **信任闭环**（现在）：修隐私 bug，让产品承诺和代码行为一致。
2. **价值闭环**（下一步）：让用户从"读每日笔记"到"回头看觉得有用"。这个没验证之前，不规划后面的东西。

### 值得押注的方向

如果价值闭环跑通了，最值得押注的是 agent 记忆层（竞品调研里的"最高匹配度"方向）。Basic Memory / IWE / Remember.md 都是从对话或笔记开始构建记忆；KeyPulse 的独特优势是能从 OS 活动自动采集工作上下文。这个差异化是真实存在的，不是营销话术。

但前提是先证明"自动采集的工作上下文"真的比"用户主动写的笔记"更有用。这个假设需要验证。

### 商业模式暂时不重要

在价值闭环跑通之前，商业模式（local app / open-core / paid sync / enterprise）可以搁置。当前阶段唯一重要的事情是：让一小撮人觉得这个东西有用。

## 总结

方向对，品味好，隐私债务必须先清。产品最核心的风险不是做不出来，是做出来没人用——尽早让真人跑通"采集 → 每日笔记 → 反馈纠正"这个最小闭环，再决定后面的编译层和 agent 层要不要照设计文档走。
