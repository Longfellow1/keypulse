# KeyPulse 展示层重构方案

## 1. 问题定义

当前 KeyPulse 输出到 Obsidian 的内容，已经完成了“存下来”和“能回看”，但还没有完成产品真正要交付的价值。

现在的问题不在于笔记数量不够，也不在于主题抽取得不够多，而在于展示层仍然站在“存储视角”而不是“产品视角”。

用户看到的是：

- 今天生成了哪些事件卡
- 这些事件被分到了哪些 topic
- 每条内容写进了哪个文件

用户真正关心的是：

- 今天哪些输入和操作根本不重要，可以忽略
- 被保留下来的内容里，哪些最值得看
- 为什么这些内容值得看
- 哪些内容只是一次性记录，哪些开始长成可复用知识

因此，展示层的第一性原则必须调整为：

**先过滤不重要的信息，再对剩余候选做排序和解释。**

不是先问“LLM 能不能帮用户抓出所有高价值话题”，而是先问“系统能不能稳定地帮用户把噪音过滤掉”。


## 2. 设计原则

这次重构只围绕三条原则展开。

### 2.1 Filter-first

先做减法，再做提炼。

系统首先要回答的是“什么不重要”，而不是“什么最重要”。

原因很简单：

- 用户来自不同行业、角色、场景
- 高价值的业务语义差异很大
- 但低价值噪音的形态反而更稳定、更通用

因此，过滤层应优先依赖规则、统计和轻量策略，而不是一开始就让 LLM 理解全部上下文。

### 2.2 Candidate-second

过滤之后剩下的内容，不直接定义为“知识”或“主题”，而是定义为“候选片段”。

候选片段只回答一件事：

**它值不值得进入用户视野。**

此时的目标不是命名，不是包装，不是讲故事，而是做稳定排序。

### 2.3 Topic-last

主题是展示结果，不是底层真相。

系统应先有：

- 过滤后的候选片段
- 每条候选片段的打分
- 每条候选片段的证据
- 候选片段之间的重复关系

然后才做：

- 聚类
- 主题命名
- 主题解释
- 首页展示


## 3. 总体流程

KeyPulse 的展示链路应固定为：

`Raw events -> Noise filter -> Candidate scoring -> Theme clustering -> LLM naming/explanation -> Obsidian surface`

其中：

- `Raw events` 是完整原始事实层
- `Noise filter` 决定系统是否可靠
- `Candidate scoring` 决定用户今天看什么
- `Theme clustering` 决定哪些内容正在长成主题
- `LLM naming/explanation` 决定表达质量，而不是决定底层真相
- `Obsidian surface` 决定用户是否真的愿意长期使用


## 4. 过滤规则层

### 4.1 目标

过滤规则层的目标不是“删数据”，而是“降低用户视野中的噪音”。

底层原始记录仍然保留，用于回溯和调试。  
过滤只影响：

- 哪些内容进入候选队列
- 哪些内容出现在首页
- 哪些内容需要送给 LLM

### 4.2 应优先过滤的噪音类型

以下内容默认不应该优先进入用户前台：

- 纯 idle、heartbeat、焦点切换类事件
- 只有窗口名，没有正文和思考痕迹的记录
- 短时间内高重复、低变化的连续事件
- 只有“发生了什么”，没有“表达了什么”的片段
- 明显不可复用的一次性碎片
- 长度极短、信息密度很低的文本
- 同类窗口或终端切换中无新增内容的重复采样

### 4.3 过滤规则的建议维度

过滤优先采用可解释规则，不直接依赖大模型。

建议至少包含这些维度：

- `重复度`
  同一时间段内是否高频重复、但无实质新增信息
- `显式输入度`
  是否来自手动保存、长剪贴板、结构化文本，而不是被动采样
- `内容完整度`
  是否具有可读正文，而不是只有标题或应用名
- `信号密度`
  是否包含明确的判断、问题、方案、约束、结论
- `可迁移性`
  是否具备未来复用的可能
- `新信息贡献`
  相比近似片段，是否带来新增信息

### 4.4 过滤层对用户的价值

过滤层不能只在系统内部生效，还要对用户可见。

用户应该能知道：

- 今天系统过滤掉了多少内容
- 过滤掉的主要原因是什么
- 是“低信号”还是“高重复”

这件事很重要，因为产品信任感来自“解释为什么没保留”，而不只是“解释为什么保留了”。


## 5. 候选打分层

### 5.1 目标

候选打分层不是为了证明“这就是知识”，而是为了稳定地做排序。

它回答的是：

**在已经通过过滤的内容里，今天最值得用户先看哪几条。**

### 5.2 候选片段的定义

候选片段是最小可归档的高信号单位。

它可以是一段手动输入，也可以是一条长剪贴板，也可以是一段明确的工作片段，但必须满足至少一项：

- 包含判断
- 包含问题
- 包含方案
- 包含可复用表达
- 包含对未来任务有影响的约束

### 5.3 建议的打分维度

候选排序建议采用规则打分，而不是先交给 LLM 判断。

核心维度可以保持精简：

- `explicitness`
  是否为明确输入，而不是被动采样
- `novelty`
  是否带来新信息
- `reusability`
  是否具备复用价值
- `decision_signal`
  是否含有判断、取舍、方向
- `recurrence`
  是否反复出现，说明不是偶发噪音
- `density`
  单位文本内的信息密度是否足够高

最终不需要一开始就区分太细的行业类型，只需要形成：

- 候选分数
- 入选原因
- 证据摘录

### 5.4 与 migration_package 的关系

这一层可以直接借鉴 migration_package 的方法论，而不是照搬具体实现。

可直接吸收的框架有四点：

- `分层筛种子`
  先用规则挑出值得进入候选层的片段，而不是全量送上去
- `限制上下文规模`
  只让高信号候选进入后续步骤，控制成本和噪音
- `合并去重再排序`
  规则结果和后续 LLM 结果都要统一回到同一个候选队列里排序
- `引入新鲜度`
  不只看“这条内容好不好”，还看“是不是已经反复讲过”

这意味着 KeyPulse 的候选层应先是一个“可排序、可解释、可去重”的队列，而不是一堆自动命名的 topic。


## 6. Obsidian 首页展示层

### 6.1 目标

Obsidian 首页不应再是“导出目录入口”，而应成为“用户今天该先看什么”的界面。

展示层不是把系统内部结构完整暴露给用户，而是把最有决策价值的结果组织出来。

### 6.2 首页结构建议

首页建议固定为四块。

#### A. Top Signals Today

这是首页第一屏。

只展示今天最值得看的少量候选片段，例如 5 到 10 条。

每条卡片展示：

- 标题
- 分数
- 入选原因
- 证据摘录
- 来源类型

这一层只回答：

**今天什么值得看。**

#### B. Filtered Out

这一层专门展示系统做减法的结果。

不必展开全部明细，但应至少告诉用户：

- 今天过滤掉多少条
- 主要噪音来源是什么
- 哪些规则命中最多

这一层只回答：

**今天什么不重要。**

#### C. Theme Candidates

这一层展示“正在长成主题”的候选簇，而不是展示所有现成 topic 文件。

每个候选主题展示：

- 候选主题名
- 当前包含多少条候选片段
- 总体置信度
- 最新证据
- 是否建议升格

这一层只回答：

**哪些内容值得从片段升级为主题。**

#### D. Review Queue

这一层提供用户动作，而不是只提供阅读。

建议动作包括：

- 升格为主题
- 并入已有主题
- 标记为噪音
- 继续观察
- 重写摘要
- 补证据

这一层只回答：

**用户接下来该怎么处理。**

### 6.3 不应放在首页第一屏的内容

以下内容应保留在底层，但不应该占据首页核心位置：

- 全量时间线
- 全量事件列表
- 大量低信号窗口名
- 只有 topic 名称、没有证据和理由的目录
- 纯文件结构导航

这些内容适合回溯，不适合产品首页。


## 7. LLM 的正确位置

LLM 不应承担“万能理解所有行业和角色”的任务。

在这条链路里，LLM 的位置应尽量靠后，且范围收紧。

适合交给 LLM 的任务：

- 候选片段的压缩表达
- 候选主题的命名
- 候选主题的说明性摘要
- 在已有证据基础上的自然语言解释

不适合一开始就交给 LLM 的任务：

- 全量原始事件筛选
- 基础噪音过滤
- 候选排序主逻辑
- 长上下文里直接猜“哪个最重要”

这条边界很关键。  
产品稳定性来自规则层，表达质量来自模型层。


## 8. 成功标准

这次展示层重构的成功，不看生成了多少笔记，而看这四个结果：

- 用户能明显感到首页噪音下降
- 用户知道系统为什么过滤掉一部分内容
- 用户能快速找到今天最值得看的少量片段
- 用户能对候选主题做明确动作，而不是被动阅读

如果做到了这四点，KeyPulse 才从“记录系统”真正走向“信息加工产品”。


## 9. 最终结论

KeyPulse 展示层的核心任务，不是让 LLM 替用户发现所有重要话题，而是：

**先稳定过滤掉不重要的信息，再把剩下的少量高信号内容做排序、解释和升级。**

因此，展示层重构应以这条顺序为唯一主线：

`Raw events -> Noise filter -> Candidate scoring -> Theme clustering -> LLM naming/explanation -> Obsidian surface`

这是后续所有实现与调优的判断基准。
